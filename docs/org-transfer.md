# Transferring console-axi to the `akash-network` org

`console-axi` currently lives at **`baktun14/console-axi`** (a personal repo) as a
POC. If it graduates, it will move to **`akash-network/console-axi`**. This is the
checklist for that move so nothing silently breaks.

## What GitHub handles automatically

Transferring a repo sets up redirects from the old owner/name for **git** (clone/
fetch/push), the **web UI**, the **REST API** (`api.github.com/repos/...`), and
**release asset** URLs (`github.com/.../releases/...`). So existing installs and the
update check keep working *for a while* via redirect.

**But update the references promptly anyway** — a redirect breaks the moment someone
creates a new repo at the old path (`baktun14/console-axi`), and stale URLs are
confusing. Treat the redirect as a grace period, not a permanent fix.

## Slug references to update

Three are real code (each independent — shell can't import TS, and the skill is a
static string), the rest derive from them or are docs. Find them all with:

```bash
git grep -n "baktun14/console-axi"
```

**Code (change these):**
- `src/version.ts` — `REPO_SLUG` → drives the update-check + `upgrade` GitHub API calls (releases/latest, asset downloads).
- `install.sh` — `REPO=` → drives release-asset downloads (and the two comment examples at the top).
- `src/skill/skill-content.ts` — the `curl … | sh` install URL embedded in the skill. **After editing, run `npm run gen:skill`** to regenerate `skills/console-axi/SKILL.md` (don't hand-edit the generated file).

**Derived / docs:**
- `skills/console-axi/SKILL.md` — regenerated by `gen:skill` (see above).
- `README.md` — install one-liner.
- `package.json` — `repository.url`, `homepage`, `bugs`.
- `docs/akash-skill/*` — update to the org URL when you re-open the upstream PR (see below).

## Release / install continuity

- **Existing installs**: their daily update check + `console-axi upgrade` hit
  `api.github.com/repos/baktun14/console-axi/...`, which redirects to the org repo
  after transfer. Ship a release from the new slug and cut a version bump so users
  pick up a binary whose baked `REPO_SLUG` points at the org (older binaries keep
  working via redirect until the old path is reclaimed).
- **`release-please.yml`**: uses the default `GITHUB_TOKEN`, so it needs no secret
  changes after transfer. Under the org's Actions policy, confirm `contents: write` is
  allowed **and** "Allow GitHub Actions to create and approve pull requests" is enabled
  (release-please needs it to open the release PR). akash-network/akash-skill already
  runs release-please, so the org policy should already permit this.
- **npm** (only if you publish there): the package name `console-axi` is unscoped and
  doesn't have to change. If you'd rather scope it to `@akashnetwork/console-axi`,
  that's a separate rename (update `name` + `bin` + install docs).

## Re-open the Akash skill contribution

The upstream contribution to `akash-network/akash-skill` was intentionally **deferred**
(personal-repo install URLs shouldn't live in the official skill). When console-axi is
in the org and validated:

1. Update the install URL in [`akash-skill/axi-cli.md`](./akash-skill/axi-cli.md) to
   the org slug.
2. Follow [`akash-skill/PR-NOTES.md`](./akash-skill/PR-NOTES.md) to add `axi-cli.md`
   under the Console-API deploy method and wire up the selector/overview edits.
3. Open the PR against `akash-network/akash-skill` (a prior version was opened and
   closed as `akash-network/akash-skill#26` — reference it).
